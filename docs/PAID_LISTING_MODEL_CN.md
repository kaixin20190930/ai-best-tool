# 付费入驻与订阅模型

这份文档定义了 AI Best Tool 的付费提交、前排展示，以及未来如果要做订阅时应该怎么设计和运营。

它主要回答三件事：

- 现在项目里已经支持什么
- 当前运营规则应该怎么定
- 将来如果要做订阅，应该在什么阶段、以什么方式做

## 1. 当前状态

目前项目支持的是 **一次性付费**，还**没有**真正的订阅制。

已经实现的能力：

- 免费提交
- 优先审核
- 固定天数前排展示
- 启动套餐
- Stripe Checkout 一次性支付
- Stripe webhook 回调确认
- 支付回调日志
- 管理员通知

还没有实现的能力：

- 订阅扣费
- 月付 / 年付自动续费
- 订阅失败后的宽限期
- 订阅取消 / 到期 / 降级生命周期
- 每月前排额度、持续曝光额度等 recurring 权益

## 2. 推荐商业化策略

当前更适合的策略是：

### 第一阶段：先用一次性收费

在目录站还处于增长期时，优先采用一次性收费。

原因：

- 用户更容易理解
- 购买阻力更低
- 不容易显得“持续收钱”
- 更适合发布期、活动期、曝光窗口这种需求
- 运营复杂度远低于订阅制

### 第二阶段：只有在“持续价值”成立时再做订阅

不要为了把一次性收费“换个壳”就硬上订阅。

订阅应该只用于持续性价值，比如：

- 开发者认证
- 条目持续维护
- 月度前排额度
- 多工具管理
- 点击 / 曝光 / 跳转分析
- 更快的修改 SLA
- 优先支持

## 3. 当前建议的一次性收费产品

### 免费提交

- 价格：`$0`
- 审核时效：`3-7 个工作日`
- 适用场景：广覆盖、首次提交

### 优先审核

- 价格：`$29 一次性`
- 审核时效：`1-3 个工作日`
- 适用场景：上线期、更新时间敏感

### 前排展示

- `3 天前排`：`$29`
- `7 天前排`：`$49`
- `14 天前排`：`$79`

适用场景：

- 发布窗口
- 短期曝光活动
- 上线宣传期

### 启动套餐

- 价格：`$99`
- 包含：
  - 优先审核
  - 14 天前排展示

适用场景：

- 一次性打包，适合上线周

## 4. 未来可考虑的订阅产品

只有在一次性付费需求被验证后，再考虑订阅。

建议触发条件：

- 每月已有 `10-20` 个稳定付费开发者
- 开发者开始反复提出“持续曝光 / 持续维护”需求
- 你已经能提供真正的 recurring 服务

推荐订阅结构：

### Verified Developer

- 价格：`$29 / 月`
- 权益：
  - 开发者认证标识
  - 更快的资料修改
  - 条目纠错优先处理
  - 单个开发者主页

### Growth Listing

- 价格：`$79 / 月`
- 权益：
  - 包含 Verified Developer
  - 每月 1 次前排额度
  - 最多 `3` 个托管工具条目
  - 更快更新 SLA
  - 基础数据报表

### Portfolio / Agency

- 价格：`$199 / 月`
- 权益：
  - 包含 Growth Listing
  - 多个工具条目管理
  - 每月前排额度
  - 更快的人工支持
  - 更偏运营的协助

这些是产品建议，不是当前已实现的计费逻辑。

## 5. 付费提交的规则

**付费提交不等于立即发布。**

付费提交的含义应该是：

- 用户为审核时效和曝光权益付费
- 工具仍然需要满足公开目录的最低质量门槛

### 最低发布门槛

一个付费工具要公开发布，至少应满足：

- 名称
- 可访问官网
- 已分配分类
- 可用 logo
- 至少一张截图或缩略图
- 能读懂的简介
- 可用的 detail 文案
- 定价标签
- 基础标签

如果缺失这些内容：

- 工具可以继续停留在 `pending review`
- 用户的付费权益保留
- 但不应该直接被公开推广

## 6. 资料优化与复查频率

### 初次审核时

在第一次审核里，应尽量完成：

- 核验官网
- 修明显缺失的元数据
- 统一分类和定价
- 优化简介和 detail 文案
- 补 media，或者要求提交者补 media

### 发布之后

建议的复查频率：

- 活跃 / 付费 / 高流量工具：每 `30 天`
- 普通已发布工具：每 `60 天`
- 长尾工具：每 `60-90 天`

### 什么情况下要优化描述

当描述：

- 太泛
- 太像生硬营销文案
- 说不清真实用途
- 没有体现工作流、定价或适用场景

就应该优化。

## 7. 前排展示规则

### 当前代码里的行为

当前实现已经改成：**前排展示从正式发布开始计时，而不是从付款成功开始计时。**

也就是：

- 付款成功：锁定权益
- 审核通过：开始计时
- 到期后：自动退出前排

### 推荐的运营规则

从公平性上，最合理的规则就是：

- 付款成功：锁定权益
- 审核通过：开始计时
- 到期后：自动退出前排

也就是说，**前排时间应从发布开始算，不应从付款成功开始算。**

### 什么情况下退出前排

工具应在以下情况退出前排：

- 购买的前排天数到期
- 工具最终未通过审核或未公开发布
- 工具违反政策或存在风险
- 官网失效或明显不可用
- 条目质量太差，不适合继续做推广位

## 8. 归档、下架、删除规则

### 归档

适用于：

- 工具质量不高，但未来还有机会补齐
- 信息过时，不值得当前继续推荐
- 不再适合公开排名，但内部记录仍有价值

### 删除 / 下架

适用于：

- 官网永久失效
- 已确认是重复工具
- 条目有误导性、风险或欺诈
- 对外对内都没有价值

### 付费工具被归档或移除时

- 不应继续保留前排状态
- 应记录原因
- 必要时通过人工决定是否补偿、退款或给 credit

## 9. 退款与补偿建议

当前项目还没有自动退款能力。

建议规则：

- 如果团队因为用户难以预见的原因拒绝付费条目，可以考虑退款或人工补偿 credit
- 如果前排窗口已经实质运行，不建议自动退款
- 欺诈、重复、明显违规等情况走人工判断

## 10. 如果未来增加订阅

如果以后增加 recurring billing，订阅生命周期必须明确。

建议状态：

- `active`
- `past_due`
- `grace_period`
- `canceled`
- `expired`

建议规则：

- 扣费失败进入 `past_due`
- 给一个 `7 天` 左右的宽限期
- 宽限期结束后，移除订阅专属权益
- 如果工具本身仍符合质量要求，不应该默认整条 listing 下架
- 优先移除“订阅权益”，而不是直接移除工具本身

### 订阅专属权益应该能独立关闭

例如：

- 认证标识
- 每月前排额度
- 加急更新 SLA
- 多 listing 配额
- 私有分析能力

这样才能把“一次性曝光”和“持续订阅服务”区分开。

## 11. 推荐的产品路线

### 现在

- 保持一次性收费为主
- Stripe 继续跑一次性支付
- 优先修正付费提交的公平性
- 收紧付费工具的最低发布门槛

### 下一步

- 把前排计时改成“发布后开始”
- 在后台更显眼地提示缺失资料
- 明确人工退款 / 补偿口径

### 更后面

- 只有在 recurring demand 被验证后，再上线订阅
- 先做好订阅生命周期，再卖订阅

## 12. 最终建议

当前最合适的商业模式是：

- 免费提交继续保留
- 优先审核继续保持一次性收费
- 前排展示继续保持固定天数窗口
- 订阅作为未来扩展，而不是现在的默认收费方式

这样既能开始变现，也不会让目录站一下子变成一个“重销售、重续费”的产品。
