# 决策摘要填写手册（中文）

这份手册对应工具详情页里的 `features.decision` 字段。

它的目标不是把工具介绍写得更长，而是把工具页写得更像“决策页”：

- 用户能快速判断适不适合自己
- 页面信息更有可信度
- 同类工具之间更容易比较
- 前台内容风格保持统一

---

## 这套字段是做什么的

当前工具详情页会优先读取这些字段：

- `compareAxes`
- `officialSummary`
- `freshnessSummary`
- `pricingSummary`
- `mediaSummary`
- `communitySummary`

如果不填，前台会回退到默认逻辑。  
如果填得好，页面会明显更像真实选型页面，而不是只有简介和跳转按钮。

---

## 字段结构

```json
{
  "features": {
    "decision": {
      "compareAxes": {
        "en": ["Model quality and speed", "Memory and workspace fit", "Ecosystem breadth"],
        "zh": ["模型质量与速度", "记忆与工作区适配", "生态广度"]
      },
      "officialSummary": {
        "en": "The official destination is stable and widely recognized...",
        "zh": "官方入口稳定且辨识度很高..."
      },
      "freshnessSummary": {
        "en": "Small product changes happen often...",
        "zh": "产品细节变化会比较频繁..."
      },
      "pricingSummary": {
        "en": "The practical pricing question is less about...",
        "zh": "实际的定价判断重点，不只是..."
      },
      "mediaSummary": {
        "en": "Screenshots help with interface expectations...",
        "zh": "截图更适合帮助你建立界面预期..."
      },
      "communitySummary": {
        "en": "User feedback tends to be broad and mixed...",
        "zh": "这类工具的反馈通常很多也很杂..."
      }
    }
  }
}
```

## 每个字段怎么写

### 1. `compareAxes`

这是“应该怎么比”的摘要，不是功能列表。

推荐写法：

- 控制在 3 条
- 每条 4 到 10 个字的中文短语更好
- 写真正影响决策的维度
- 不写空泛词，比如“功能强大”“体验很好”

适合写：

- 模型质量与速度
- SEO brief 工作流
- 数据覆盖广度
- 团队协作适配
- 自动化连接能力

不适合写：

- 界面好看
- 很受欢迎
- 很智能

---

### 2. `officialSummary`

这是“官网可信度和官网在决策里扮演什么角色”的摘要。

它不是让你夸官网，而是帮助用户判断：

- 这类工具是不是官方站点本身已经足够可信
- 这页存在的价值是不是“帮助选型”，而不是“帮你确认真假”

推荐长度：

- 中文 1 句
- 25 到 60 字

适合写：

- 官网是否稳定
- 官网是否容易验证
- 本页更适合做什么判断

---

### 3. `freshnessSummary`

这是“这条工具信息多久可能变”的判断。

核心不是说“我们更新很勤”，而是告诉用户：

- 哪些信息相对稳定
- 哪些信息最好回官网复核

推荐长度：

- 中文 1 句
- 25 到 70 字

适合写：

- 套餐边界变化快不快
- 产品界面变化快不快
- 能力边界是否经常调整

---

### 4. `pricingSummary`

这是“真正影响付费决策的点是什么”。

不要只重复“有免费版 / 有付费版”。  
要回答的是：

- 什么情况下值得付费
- 付费价值来自哪里
- 用户从免费路径升级的触发点是什么

推荐长度：

- 中文 1 句
- 25 到 70 字

适合写：

- 高频使用才有价值
- 团队协作时价值上升
- 深度研究时更值得付费
- 内容工作流完整后才体现 ROI

---

### 5. `mediaSummary`

这是“截图、logo、界面预览对这个工具到底有没有决策意义”。

有些工具很依赖界面密度，有些工具截图意义不大。  
这一段是帮用户建立正确预期。

推荐长度：

- 中文 1 句
- 20 到 60 字

适合写：

- 是否需要通过截图判断信息密度
- 是否需要通过预览判断工作流复杂度
- 截图是帮助理解界面，还是只是辅助参考

---

### 6. `communitySummary`

这是“评论和社区反馈该怎么看”。

不是简单说“用户反馈很好”，而是告诉用户：

- 哪类评论最有参考价值
- 哪类热度信号不要过度解读

推荐长度：

- 中文 1 句
- 25 到 70 字

适合写：

- 应优先看真实工作流反馈
- 应优先看重度用户评论
- 热度高不等于适合
- 要看评论是否对应你的场景

## 统一写作规则

### 要这样写

- 以“帮助用户做判断”为中心
- 写决策逻辑，不写营销口号
- 用平实、可信、克制的语气
- 承认不确定性，比如价格和能力边界可能变化
- 让不同工具的文风尽量统一

### 不要这样写

- 不写“全球领先”“革命性”“最强”
- 不写明显站队式评价
- 不写空洞结论
- 不把官网介绍改写一遍
- 不把这一段写成广告文案

---

## 不同类型工具的写法重点

### 写作类工具

优先强调：

- 长文质量
- 起草还是编辑
- 语气稳定性
- 适合个人还是团队

### Web3 / 数据分析类工具

优先强调：

- 数据覆盖广度
- 查询深度
- 仪表盘复用
- 是否适合研究型工作流

### 开发类工具

优先强调：

- 编辑器内工作流
- 多文件上下文
- 真实实现速度
- 对重构和调试是否有帮助

### 自动化类工具

优先强调：

- 连接器生态
- 编排复杂度
- 团队复用能力
- 是否容易落地到真实流程

## 什么时候需要补这套字段

优先补这几类：

1. 搜索需求强的工具
2. 某个分类的锚点工具
3. 你准备重点运营或推广的工具
4. 有付费入驻或商业价值的工具
5. 已经有一些流量，但详情页还很薄的工具

不建议一开始给所有工具都补。  
更好的节奏是：每周补 5 到 10 个最重要的页面。

---

## 什么时候需要更新

建议按下面节奏复查：

- 高流量工具：每月复查一次
- 付费展示工具：上线前复查一次，展示期内至少复查一次
- 快速变化的 AI 工具：1 到 2 个月复查一次
- 相对稳定的基础设施或数据工具：按季度复查

如果出现下面情况，也应该提前更新：

- 定价模型大改
- 官网改版明显
- 产品定位变化
- 主要使用场景变化
- 评论区出现大量一致性反馈

## 最小验收标准

一条工具如果补了 `features.decision`，至少要满足：

- `compareAxes` 有 3 条真实可比维度
- 其余 5 个 summary 都不是空话
- 中文表达自然，不像机器堆砌
- 能帮助用户回答“我该不该试这个工具”

如果只是把默认文案换个说法，不如先不填。

---

## 一个简单判断法

写完之后，问自己 3 个问题：

1. 用户看完会不会更容易决定要不要点官网？
2. 这段话是不是在帮助判断，而不是在做宣传？
3. 把工具名遮住后，这段话是否仍然有明确场景，而不是套话？

如果这 3 个问题都能过，这条内容通常就够用了。
